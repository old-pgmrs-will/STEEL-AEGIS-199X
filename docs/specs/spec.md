# 縦スクロールシューティングゲーム仕様書

## 1. 文書目的

本書は、ブラウザで動作する1990年代王道系の縦スクロールシューティングゲームを企画・設計するための仕様書です。  
目的は、ゲームデザイン、演出、ビジュアル制作、実装判断、テスト観点を一つの文書に統合し、後続の開発者やアーティストが追加説明なしで制作に着手できる状態を作ることです。

## 2. 企画概要

### 2.1 コンセプト

- 1990年代アーケード系縦スクロールシューティングの文法を現代ブラウザで再構成する
- スプライト主体の緻密なビジュアルと、見て避けて撃ち込む快感を重視する
- 初回プレイでも理解しやすく、繰り返しプレイでスコアと攻略精度が伸びる設計にする

### 2.2 企画キーワード

- 王道
- 密度感のあるスプライト表現
- 多彩な敵編成
- ステージ終盤ボス戦
- 攻撃力・防御力の成長
- パターン学習
- リプレイ性

### 2.3 想定タイトル

- 仮題: `STEEL AEGIS 199X`

### 2.4 プレイヤー体験目標

- 開始30秒で「大量の敵を撃ち落とす爽快感」を理解できる
- 1ステージの中で通常戦闘、地形演出、中型機、ボス戦という緩急を体感できる
- パワーアップ取得により、見た目と戦術の両方が段階的に強化される
- ミスをしても即終了ではなく、復帰可能性を残しつつ緊張感を維持する

## 3. 対応環境と前提

### 3.1 対応プラットフォーム

- PCブラウザを主対象とする
- 対応ブラウザ: 最新の Chrome、Edge、Safari、Firefox
- スマートフォン閲覧は補助対応とし、正式なプレイ対象は物理キーボードまたはゲームパッドを利用できる環境とする

### 3.2 表示仕様

- 基準解像度: 1080 x 1920 相当の縦画面をゲーム内部基準とする
- 実表示はレターボックス方式で比率維持
- UI を含む描画はスケーリング対応し、低解像度でも視認性を損なわない

### 3.3 セッション設計

- 1プレイ全5ステージ
- 1周クリア時間目安: 25分から35分
- コンティニューあり
- ブラウザリロードやタブ移動時はポーズ扱い

## 4. ゲームの柱

### 4.1 主要要素

1. 通常ショットとサブウェポンで敵編隊を処理する撃ち込み
2. 敵弾密度を見切る回避
3. アイテム回収による攻撃力・防御力の強化
4. 各ステージ終盤ボスのパターン攻略
5. スコア、ノーミス、回収率を高める再挑戦

### 4.2 1990年代王道シューティングとして守る要件

- ステージごとに明確なテーマと中ボス、ボスを置く
- 敵編隊には暗記可能な進入パターンと、咄嗟の対応が必要な変則編成を混在させる
- 1つの武装を延々強化するだけでなく、取得順と死亡時の復旧が攻略に影響する
- ボムは緊急回避と攻撃補助を兼ねる
- スコアは単なる撃破点ではなく、危険行動や回収行動を評価する

## 6. 操作仕様

### 6.1 基本操作

- 移動: `Arrow Keys` または `WASD`
- ショット: `Z` またはゲームパッド主ボタン
- ボム: `X`
- 低速移動: `Left Shift`
- ポーズ: `Esc`

### 6.2 操作方針

- ショットは押しっぱなし連射可能
- 低速移動中は移動速度を下げ、被弾判定表示を有効化する
- ゲームパッドではアナログスティックと十字キーの両方を許可する

## 7. プレイヤー機体仕様

### 7.1 機体構成

- 初期実装は3機体
- 各機体は移動速度、正面火力、拡散性能、防御寄り性能が異なる

### 7.2 機体案

| 機体 | 役割 | 特徴 |
| --- | --- | --- |
| Type-A `Valk Lance` | 標準型 | 正面火力とバランスに優れる |
| Type-B `Iris Gail` | 高機動型 | 移動が速く、広範囲攻撃向き |
| Type-C `Bulwark Note` | 重装型 | 防御性能と追尾補助に優れる |

### 7.3 プレイヤーステータス

- 残機
- シールド耐久値
- ショットレベル
- サブウェポンレベル
- ボムストック
- スコア
- コンボ倍率

### 7.4 被弾と復帰

- 通常被弾時はシールドを消費
- シールドが0の状態で被弾すると残機を失う
- ミス後は短時間無敵でその場復帰
- 復帰時に一部火力を失うが、完全初期化はしない

## 8. 攻撃システム

### 8.1 メインショット

- 自動連射の主武装
- レベル1からレベル5まで段階強化
- 強化で弾数、貫通性、発射幅、エフェクトが変化する

### 8.2 サブウェポン

- 機体ごとに固有
- 例:
  - Type-A: 直進レーザーオプション
  - Type-B: 扇状ミサイル
  - Type-C: 追尾ビット

### 8.3 チャージ要素

- 王道感を崩さない範囲で短押し主体とする
- 低速移動を一定時間継続した場合のみ補助チャージが有効化される
- 補助チャージは強力な一撃ではなく、防御を兼ねた短時間前方バリア射撃とする

### 8.4 ボム

- 全画面または広範囲攻撃
- 弾消し効果あり
- 使用中は短時間無敵
- ボムの使用はスコア上不利にならず、抱え落ち回避を推奨する設計とする

## 9. 防御・成長システム

### 9.1 攻撃力パワーアップ

- 赤アイテム: メインショット強化
- 青アイテム: サブウェポン強化
- 金アイテム: 一時的攻撃ブースト

### 9.2 防御力パワーアップ

- 緑アイテム: シールド最大値拡張
- 白アイテム: シールド即時回復
- 紫アイテム: 一定時間被弾軽減

### 9.3 取得設計

- 敵撃破、特定地形破壊、隠し配置から出現
- 同種アイテム連続取得で効果効率を少し上げる
- ミス時は一部アイテムを周囲に再放出し、再回収で戦線復帰しやすくする

### 9.4 成長の見た目反映

- ショット強化時は弾色、発射口エフェクト、オプション数を変える
- 防御強化時は機体周囲のシールド輪郭や発光レイヤーを変える
- 強化の視覚差分はプレイヤーと観戦者が即座に理解できることを必須要件とする

## 10. スコアシステム

### 10.1 加点要素

- 敵撃破点
- 連続撃破コンボ
- 接近撃破ボーナス
- 弾幕かすり
- アイテム回収率
- ノーミス、ノーボム、ステージクリアボーナス

### 10.2 コンボ仕様

- 一定時間内に連続撃破でコンボ継続
- 大型敵撃破はコンボ継続時間を延長
- ミスまたは一定時間撃破なしでコンボ終了

### 10.3 スコア設計意図

- 生存だけでなく、危険地帯への踏み込みを評価する
- 上級者はスコアプレイ、初級者はクリア重視で遊べるよう両立させる

## 11. ステージ構成

### 11.1 全体構成

- ステージ数: 5
- 各ステージにテーマ、専用雑魚編成、中型機、ボスを設定
- 難度は段階的に上げるが、ステージ1からビジュアル密度と演出密度は高くする

### 11.2 ステージ一覧

| ステージ | テーマ | 主な敵傾向 | ボス性質 |
| --- | --- | --- | --- |
| 1 | 沿岸工業地帯 | 直進小型機、固定砲台、輸送艦 | 基本パターン学習用 |
| 2 | 雲海上空要塞 | 高速編隊、空中機雷、旋回ドローン | 横薙ぎ弾幕と部位破壊 |
| 3 | 砂漠機甲戦線 | 地上戦車、列車砲、埋設砲台 | 長期戦寄りの装甲型 |
| 4 | 極圏エネルギー施設 | 追尾機、反射弾ユニット、レーザー塔 | 画面制圧型弾幕 |
| 5 | 軌道上司令中枢 | 混成精鋭、自己修復兵装、親衛機 | 多段階変形の最終ボス |

### 11.3 ステージ尺

- 通常戦闘: 3分から4分
- 中盤山場: 1分
- ボス戦: 2分から3分

## 12. 敵キャラクター仕様

### 12.1 敵カテゴリ

1. 小型機: 大量出現、低耐久、編隊で見せる
2. 中型機: 弾幕起点、アイテム源、戦況の節目を作る
3. 大型兵器: 地形と一体化した見せ場担当
4. 固定砲台: 進行ルート制御と画面圧力担当
5. 特殊機: 回復阻害、シールド展開、弾速変化などのギミック担当

### 12.2 敵バリエーション設計

- 1ステージあたり最低8種の雑魚敵を用意する
- 5ステージ通算で40種以上の敵バリエーションを確保する
- 色替えだけではなく、進入方向、攻撃間隔、撃破時挙動を差別化する

### 12.3 敵行動パターン

- 直進突撃
- 画面端からの曲線侵入
- 停止射撃後の離脱
- プレイヤー追尾
- 自爆接近
- 地形陰からの奇襲
- 護衛対象の周回防衛

### 12.4 中型敵の役割

- ステージごとに2体から4体
- 高耐久と局所弾幕でプレイヤーに位置取り変更を強制する
- 撃破報酬として高確率でパワーアップを落とす

## 13. ボス仕様

### 13.1 共通要件

- 各ステージ終端に1体配置
- 登場演出、ボス専用BGM、専用HPバーを持つ
- 形態変化または攻撃フェーズ変化を最低2段階持つ
- 撃破時は大爆発、リザルト演出、次ステージへの転換を入れる

### 13.2 ボス設計原則

- 初見で圧倒し、再挑戦で攻略可能になる可読性を持たせる
- 安置一択ではなく、リスクと見返りのある安全地帯移動を設計する
- 部位破壊を導入し、スコアや難度に差を付ける

### 13.3 ボス案

| ステージ | ボス名 | 特徴 |
| --- | --- | --- |
| 1 | `GUNBARGE HECTOR` | 両舷砲台と中央主砲を持つ大型戦艦 |
| 2 | `SERAPHIM FORT-β` | 雲中から出現する浮遊要塞、分離攻撃あり |
| 3 | `DUNE TYRANT` | 多脚装甲列車砲、地上砲台の連携あり |
| 4 | `POLARIS REQUIEM` | 極太レーザーと反射弾を使う発電中枢兵器 |
| 5 | `THRONE OF NOVA` | 人型と要塞形態を切り替える最終司令機 |

## 14. 弾幕・難度設計

### 14.1 難度モード

- `Novice`
- `Arcade`
- `Expert`

### 14.2 調整対象

- 敵弾速度
- 敵弾密度
- アイテム出現率
- 敵耐久値
- コンボ維持猶予

### 14.3 動的難度補正

- 連続被弾時は短時間だけアイテム出現率を上げる
- 上級者向けに、ノーミス継続時は一部配置が強化される内部ランクを採用する
- 内部ランクは露骨な理不尽増加を避け、演出差よりも密度差に留める

## 15. UI/UX 仕様

### 15.1 HUD 要素

- 残機
- シールドゲージ
- ショットレベル
- サブウェポンレベル
- ボム数
- スコア
- コンボ
- ステージ進行度
- ボスHP

### 15.2 画面設計

- ゲームプレイ領域を中央縦長に固定
- 左右サイドにスコア、機体情報、アート演出帯を配置
- 情報量は多くても、プレイ領域に干渉しない

### 15.3 利便機能

- キーコンフィグ
- ゲームパッド表示切替
- リトライ
- オートポーズ
- 音量個別調整
- 点滅軽減オプション

### 15.4 UI画像生成ワークフロー

- UI は「レイアウト設計」と「装飾素材制作」を分離して設計する
- 数値、ラベル、ゲージ目盛り、ボタン文言など可読性を最優先する要素は、画像生成に依存せず実装側で描画または組版する
- 画像生成は、左右サイドパネル、警告帯、機体情報プレート、ボスHP枠、装飾フレーム、背景演出帯、アイコン原画などの視覚演出素材に限定して使う
- 生成画像は完成UIとして直接使用せず、最終的に分割可能な部材原画として扱う

### 15.5 UI画像生成の実制作手順

1. ワイヤーフレームを作成し、HUD 情報の位置、サイズ、優先順位を先に固定する
2. 固定済みレイアウトに対して、必要な装飾パーツの一覧を定義する
3. パーツ単位で画像生成を行い、UI の世界観、色調、金属感、発光表現を揃える
4. 生成物から角飾り、フレーム、帯、エンブレム、アイコンなどを切り出す
5. 切り出し後の素材を透過化し、輪郭修正、不要ディテール削除、縮小耐性確認を行う
6. 実装時は 9-slice 相当の伸縮やスプライト差し替えで、解像度差や状態差分に対応する

### 15.6 生成対象と非生成対象

| 区分 | 対象 | 方針 |
| --- | --- | --- |
| 生成対象 | フレーム、警告演出帯、装飾パネル、機体アイコン原画、メダル風バッジ、背景演出帯 | 画像生成でベースを作り、後工程でパーツ分離する |
| 半生成対象 | ボムアイコン、残機アイコン、難度バッジ、ステージエンブレム | 画像生成で意匠案を出し、最終的に縮小前提で手修正する |
| 非生成対象 | スコア数字、コンボ値、HP 数値、本文ラベル、操作説明文字列、細かな目盛り | フォント描画またはベクター／手組みで実装する |

### 15.7 UI状態差分の扱い

- 通常、選択中、警告中、被弾時、ボス戦、無効状態の6系統を基本状態とする
- 色変化や発光量変更で済む差分は実装側シェーダーまたは色替えで対応する
- 形状変化が必要な差分だけ、別スプライトとして追加生成または手修正する
- 点滅表現はアクセシビリティ配慮のため、設定で弱められる前提で設計する

### 15.8 UI品質基準

- プレイ中に最も重要な情報である残機、シールド、ボム、ボスHP は一瞥で判読できること
- 生成装飾が主張しすぎて、数値やゲージを読みづらくしないこと
- 1920px 基準だけでなく縮小表示でも視認性を維持できること
- ステージごとの演出差分があっても HUD の情報配置は固定し、学習コストを増やさないこと

## 16. ビジュアル仕様

### 16.1 全体アート方向

- 1990年代アーケードゲームの読みやすいシルエットを踏襲する
- 情報量は高くするが、輪郭は太めで敵味方判別を優先する
- 写実一辺倒ではなく、メカの記号性と派手な爆発表現を共存させる

### 16.2 スプライト表現要件

- プレイヤー、敵、弾、爆発、UI装飾はスプライトベースで管理する
- 1体の大型ボスは複数パーツの組み合わせスプライトで構成する
- 破壊時差分、被弾発光、エンジン噴射、影レイヤーを別パーツ化する

### 16.3 背景表現

- 1ステージにつき遠景、中景、近景の3層以上のパララックス構成
- 雲、煙、雪、砂塵、光柱などの環境エフェクトを重ねる
- 背景は派手でも敵弾と混同しない明度設計にする

## 17. 画像生成前提のアセット制作仕様

### 17.1 制作前提

- 主要ビジュアルは画像生成を起点に制作する
- 生成物をそのまま使うのではなく、スプライト用途に合わせて人手または補助ツールで整形する
- 最終的なゲーム内使用アセットは透過、輪郭修正、色数整理、アニメーション分解を前提とする

### 17.2 生成対象

- 自機3種の全身デザイン
- 雑魚敵40種以上のベース案
- 中型機10種前後
- ボス5体の全体図と部位図
- 各ステージ背景用コンセプトアート
- UI 装飾フレーム
- UI パネル、警告帯、エンブレム、アイコン原画
- 爆発、噴射、エネルギー弾のエフェクト素材

### 17.3 画像生成ガイドライン

- 真上視点から斜め前方視点まで、実ゲームで使う角度を固定してプロンプトを統一する
- 機体は「判読しやすい外形」「前方火器位置が明確」「エンジンとコックピットの位置が識別可能」であること
- 背景は横長一枚絵ではなく、タイル分解やレイヤー抽出を前提に生成する
- UI はゲーム画面の視認性を優先し、装飾過多で数値を読みにくくしない
- UI は完成画面全体を生成するのではなく、再利用可能なパーツ原画を単体生成する

### 17.4 生成後の整形工程

1. 候補選定
2. 正面・側面・攻撃差分の再生成
3. 輪郭調整と不要ディテール削除
4. スプライトサイズへの縮小検証
5. パーツ分離
6. 発光・ダメージ差分追加
7. スプライトシート化

### 17.5 アセット仕様目安

| 種別 | 推奨サイズ | 備考 |
| --- | --- | --- |
| 自機 | 128x128 | 当たり判定は見た目より小さく設定 |
| 小型敵 | 96x96 | 編隊時の視認性重視 |
| 中型敵 | 192x192 | 攻撃部位差分あり |
| ボス部位 | 256x256 から 1024x1024 | 分割前提 |
| 爆発 | 128x128 から 512x512 | 連番アニメーション化 |
| 背景タイル | 512x512 または 1024x1024 | レイヤー分割前提 |

### 17.6 ビジュアル品質基準

- 縮小後にも敵の向きと危険部位が判読できる
- 弾と背景装飾が混同しない
- ステージごとの色相が明確に異なる
- 同系統敵でもサイズと兵装シルエットで役割が分かる

## 18. 音響仕様

### 18.1 BGM 方針

- FM音源感を残しつつ現代的に厚みを出したシンセサウンド
- ステージ前半、後半、ボスで印象を切り替える

### 18.2 SE 方針

- ショット、着弾、撃破、警告音、ボム、アイテム取得に明確な差を付ける
- 強化段階ごとにショット音を変え、成長を聴覚でも伝える

## 19. 技術仕様

### 19.1 実装前提

- HTML5 Canvas または WebGL ベースの描画
- 60fps を目標
- 入力遅延を抑えるため、描画と入力処理は軽量に保つ

### 19.2 必須システム

- ステージ進行管理
- 敵出現スクリプト
- 当たり判定
- パワーアップ管理
- スコア計算
- ボスフェーズ管理
- セーブ設定管理
- リプレイまたはハイスコア保存

### 19.3 データ分離方針

- 敵定義、出現テーブル、武器性能、スコア係数はデータ駆動化する
- バランス調整値をコードに埋め込みすぎない

## 20. 演出仕様

### 20.1 ステージ演出

- ボス接近警告
- 大型兵器の画面外侵入
- 背景オブジェクトの崩壊
- ステージ終盤の密度上昇演出

### 20.2 フィードバック

- 被弾時は画面フラッシュ、音、無敵表示を明確にする
- アイテム取得時は小さな加点表示と機体発光を入れる
- ボス部位破壊時は一段階大きな演出で達成感を出す

## 21. リプレイ性と拡張要素

### 21.1 リプレイ要素

- 機体ごとの攻略差
- 難度差
- スコアアタック
- ノーミス挑戦
- 隠しアイテムと隠し敵配置

### 21.2 将来拡張候補

- 2周目高難度ルート
- ボスラッシュ
- デイリーチャレンジ
- オンラインランキング

## 22. バランス目標

- 初級者は `Novice` で1週間以内に1周クリアを目指せる
- 中級者は `Arcade` で機体ごとの差を体感できる
- 上級者は `Expert` と内部ランクで十分な詰め要素を得られる

## 23. 受け入れ基準

### 23.1 企画受け入れ基準

- 5ステージ分のテーマ、敵、ボス、成長要素が定義されている
- 攻撃力と防御力の両方に明確な強化経路がある
- 画像生成前提のアセット制作工程が定義されている
- 王道シューティングとして必要な操作、復帰、スコア、ボス設計が網羅されている

### 23.2 実装受け入れ基準

- PCブラウザで60fps目標を維持できる
- 1プレイを最初から最後まで破綻なく完走できる
- 各ステージで敵の視認性と背景判別性が保たれている
- ミス後に戦線復帰可能で、パワーアップ再構築の余地がある

## 24. 未確定事項

- コンティニュー回数制限の有無
- オンラインランキングの実装範囲
- スマートフォン向けタッチ操作の正式対応有無
- ストーリー演出量とボイス導入有無

## 25. まとめ

本企画は、1990年代の王道縦スクロールシューティングの骨格を守りながら、ブラウザ向けに遊びやすさと視認性を最適化した作品を目指します。  
特に、攻撃力と防御力の二軸成長、多彩な敵編成、ステージ終端ボス、画像生成起点の高密度スプライト制作を中核要件とし、企画から実装まで一貫した判断基準を持たせます。

## 26. 現在の実装反映状況

### 26.1 初期プレイアブル版で実装済みの項目

- `HTML5 Canvas` と `TypeScript` によるブラウザ実行基盤
- タイトル画面、プレイ中、ポーズ、ゲームオーバー、ステージクリアの基本遷移
- 自機1種による移動、ショット、ボム、低速移動
- 単一ステージのプロトタイプ進行
- 小型敵、中型相当敵、簡易ボスの出現と攻撃
- 敵弾、自機被弾、シールド減少、残機減少、無敵復帰
- ショット強化、シールド回復、ボム補給のアイテム取得
- スコア、コンボ、シールド、残機、ボム、ボスHP を含む簡易 HUD
- ピクセルデータ原稿からのスプライトシート生成と、スプライト ID 参照によるキャラクター描画
- `GPT-image-1.5` 前提のスプライト生成マニフェスト、検収チェックリスト生成、スプライトシート自動パックの補助スクリプト
- candidate 画像の背景除去とセル正規化による `approved/normalized` 出力

### 26.2 初期プレイアブル版で未実装の項目

- 機体選択
- 5ステージ全量
- 複数ボスフェーズ
- 画像生成済みスプライト差し替え
- 永続設定保存
- オンラインランキング
- 実APIキーによる `GPT-image-1.5` 生成実行の本番確認

### 26.3 現在のビジュアル実装方針

- キャラクター描画は、ピクセルデータ定義から生成したスプライトシートを参照する方式へ移行する
- ランタイムは図形直描きではなく、スプライト ID を通じて自機、敵、ボス、アイテム、弾を描画する
- 画像生成アセット投入前の期間は、外部画像の代わりにローカルで保持するピクセルデータ原稿をスプライトシート化して用いる
- 現在の初期実装では、ローカル定義のピクセルデータを起動時にスプライトシートへパックしている

### 26.4 スプライトシート運用方針

- キャラクター原稿はピクセルグリッドとパレット参照で定義する
- ビルド時または起動時にスプライトシートへパックし、各スプライトの矩形情報をメタデータとして保持する
- 実装側は `spriteId` と描画位置、拡大倍率、向きのみを指定し、ピクセル内容には直接依存しない
- 後続で外部画像生成結果を採用する場合も、同一 `spriteId` と矩形メタデータへ変換して差し替える

### 26.5 画像生成自動化方針

- スプライト画像生成は `GPT-image-1.5` を用いた個別スプライト生成で自動化する
- 生成対象は `sprite-list.csv` と prompt manifest で定義し、`spriteId` 単位で管理する
- 生成結果は `raw`、採用候補は `candidates`、検収済みは `approved` に分離して保管する
- スプライトシートと `sheet-layout.json` は `approved` 画像から機械的に生成する
- シート分割はカテゴリとセルサイズを基準に行い、異なるセルサイズは同一 PNG シートへ混在させない
- シート名は `<category>-<cellWidth>x<cellHeight>.png` を基本形とし、`enemy_small` のようなカテゴリ名はハイフン区切りへ正規化する
- 完了判定は `required = yes` の全 `spriteId` が `packed` になることとする
- API 認証エラー時は生成処理全体を停止し、作業者へ `.env.local` と権限設定の確認を促す
- レート制限や一時的サーバーエラー時は限定回数だけ自動再試行し、失敗内容は別ログへ記録する
- 実行中は対象 `spriteId`、試行回数、再試行、保存完了、失敗要約を標準出力へ逐次表示する
- `candidate` 選定後は背景除去とセル正規化を行い、`approved/normalized` へ配置してからシート化する

### 26.6 ランタイム描画反映方針

- ゲーム実装はコード内ピクセル定義を直接描画せず、`assets/sprites/sheets/*.png` と `assets/sprites/manifests/sheet-layout.json` を参照する
- 起動時にシート画像群とレイアウト JSON をロードし、描画系へまとめて渡す
- ランタイム `spriteId` は台帳上の `sprite_id` に統一する
- `sheet-layout.json` にないアンカー情報はランタイム側の既知マップで補完する
- アセット読込失敗時は起動を継続せず、エラーを利用者へ通知する
- 実装では `src/rendering/spriteAssets.ts` が `import.meta.glob` で JSON と PNG を解決し、`src/rendering/sprites.ts` が `spriteId` 単位で切り出して描画する
- 実装では `src/main.ts` がスプライト読込完了後に `Game` を開始する
- 現行プレイアブル版の自機表示は `player_type_a_idle` を利用する
- 現行プレイアブル版ではゲーム用スプライトの表示サイズを基準倍率の 2 倍で描画し、当たり判定と境界チェックもその表示サイズに追従させる
- 現行プレイアブル版では `SPRITE_SCALE_MULTIPLIER = 2` を用い、プレイヤー、敵、弾、アイテムの表示倍率と当たり判定半径を同一倍率で拡大する
