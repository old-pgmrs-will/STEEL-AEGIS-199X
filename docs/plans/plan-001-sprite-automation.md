# 実装計画

## 1. 目的とスコープ

本計画は、縦スクロールシューティングゲームの実装開始にあたり、最初のプレイアブル版を成立させるための実装範囲を定義するものです。  
今回のスコープは、ブラウザで起動できるゲーム基盤を作成し、タイトル表示、ゲーム開始、プレイヤー移動、ショット、敵出現、敵弾、当たり判定、残機とシールド管理、スコア、簡易 HUD、ゲームオーバーまでを一連で動作させるところまでとします。  
加えて、キャラクター描画アーキテクチャは、図形直描きから「画像生成起点のピクセルデータをスプライトシート化し、実装はスプライト ID 参照で描画する方式」へ変更します。

## 2. 採用アプローチと理由

### 2.1 採用アプローチ

- `TypeScript` と `Vite` でブラウザ実行基盤を作る
- 外部描画ライブラリを増やさず、初期段階は `HTML5 Canvas` で最小ゲームループを実装する
- キャラクターの見た目データは、ピクセル単位のソース定義からスプライトシートへ集約し、描画コードは図形ではなくスプライト参照に統一する
- データ駆動化を意識しつつも、初回実装では最低限のデータ定義とシンプルな構成で動作を優先する

### 2.2 理由

1. 空のリポジトリから最短でプレイ可能状態まで到達できる
2. 初期実装で複雑な描画ライブラリ依存を避け、ゲームループと仕様整合を先に固められる
3. 後続で `PixiJS` などへ移行する場合でも、ロジックと描画の責務分離を保ちやすい
4. 画像生成アセットの差し替え先をスプライトシートへ一本化できる

## 3. 却下した代替案

### 3.1 初手から `PixiJS` を導入する案

- 却下理由: 初回実装時点では依存導入、描画ラッパ、アセット管理まで同時に増え、空の状態からの立ち上がりが遅くなるため

### 3.2 まず設計だけ続けてコード実装を遅らせる案

- 却下理由: 既に企画仕様と詳細仕様が存在するため、次は動作確認できる最小実装へ進む方が検証価値が高いため

### 3.3 ECS などの抽象度の高いアーキテクチャを初手で導入する案

- 却下理由: 現段階では過剰であり、責務分離された素直な状態更新構造で十分だから

## 4. 制約と前提

- リポジトリは現時点で空に近く、初期構成から作成する必要がある
- 外部生成済み画像はまだ存在しないため、当面は画像生成想定のピクセルデータをローカル定義し、スプライトシート化して使う
- コードコメントおよびドキュメントは日本語で記述する
- ビルドまたは実行確認は、依存取得が必要な場合に `npm install` を行う前提とする
- 今回は単一ステージのプロトタイプ実装を優先し、全5ステージ実装は次フェーズとする

## 5. 実装対象

- `package.json`, `tsconfig.json`, `vite.config.ts` などの基盤ファイル
- `index.html` と `src/` 配下の初期ゲームコード
- 最小のシーン管理
- Canvas 描画ベースのゲームループ
- プレイヤー、敵、弾、アイテムの基本挙動
- HUD とゲームオーバー表示
- ピクセルデータ定義、スプライトシート生成、スプライト参照描画基盤

## 6. 今回実装しないもの

- 外部画像生成サービスとの直接連携
- 複数機体選択
- 5ステージ全量
- ボス多段階フェーズ
- 永続セーブやランキング
- オプション画面の完全実装
