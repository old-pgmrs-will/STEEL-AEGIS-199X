# 実装計画

## 1. 目的とスコープ

本計画は、ゲーム内スプライトの表示サイズを現状の 2 倍へ引き上げ、見た目の拡大に合わせて当たり判定と画面端の境界チェックも整合させることを目的とします。  
今回のスコープは、プレイヤー、敵、弾、アイテムの描画倍率、当たり判定半径、プレイヤー移動範囲、画面外消滅条件の調整、およびビルド確認までとします。

## 2. 採用アプローチと理由

### 2.1 採用アプローチ

- ゲーム用スプライトの描画倍率を定数化し、現状比 2 倍へ一括変更する
- `sheet-layout.json` のフレームサイズから描画時の見かけ半径を求める
- 当たり判定と境界チェックは、固定マジックナンバーではなくスプライト実寸ベースの補助関数経由に寄せる

### 2.2 理由

1. 拡大方針を定数管理にすると今後の再調整が容易なため
2. 見た目と当たり判定の乖離を減らせるため
3. 境界処理をスプライトサイズ基準にするとアセット差し替え後も破綻しにくいため

## 3. 却下した代替案

### 3.1 描画倍率だけを変更し、当たり判定と境界は既存の固定値のままにする案

- 却下理由: ユーザー要求に反し、見た目と挙動が一致しなくなるため

### 3.2 元画像そのものを再生成して大きくする案

- 却下理由: 今回の条件が「画像はそのまま使う」であり、表示側で解決すべきため

## 4. 制約と前提

- `sheet-layout.json` はフレーム矩形のみを持つため、半径や境界余白はランタイム側で計算する
- プレイヤーの精密判定はゲーム性上あえて小さく残す余地があるが、今回は見た目拡大に合わせて基準値も拡大する

## 5. 実装対象

- 描画倍率定数
- フレーム寸法取得 API
- 当たり判定半径の拡大
- プレイヤー移動境界の見直し
- エンティティの画面外除去境界の見直し

## 6. 今回実装しないもの

- ステージレイアウト全体の再調整
- 難易度や敵出現密度の再設計
- UI レイアウトの再スケーリング
